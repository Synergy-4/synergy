Game Design Specification

Colour Matching Game

A solid-colour card matching game for pure colour recognition. The player clears a middle board by tapping hand cards that match colours displayed in the target zone. Penalty on mismatch, scales from 10-card beginner to 100-card mastery.

Module: M02 / M23 (Colour Identification)
Mechanic: tap_to_match
Flutter widget: ColourMatchingGame
Timed: Optional per difficulty

Core Game Loop

Round Initialisation
  1. Server sends SDUI payload with game_config (pool, hand size, middle count, colour set, timer)
  2. Client shuffles the pool of colour cards
  3. Deal hand_size cards from pool top → player's hand
  4. Generate middle_count target cards. Rule: every colour in the middle MUST exist at least once in the remaining pool. Prevents unwinnable state.
  5. Start timer if timed: true
  6. Game state → PLAYING
On Each Player Tap
Pseudocodecore game loop — tap handler
onCardTapped(handCard): // 1. Check for a colour match in the middle zone matchingTargets = middleCards.where((t) => t.colour == handCard.colour) if matchingTargets.isNotEmpty: // MATCH — one tap clears ALL cards of that colour score += calculateMatchScore(matchingTargets.length, timeRemaining) middleCards.removeAll(matchingTargets) // clear from target zone playMatchAnimation(handCard, matchingTargets) // burst effect hand.remove(handCard) // remove played card drawFromPool() // immediately replace else: // MISS — card is permanently lost (penalty) errorCount++ hand.remove(handCard) // card discarded playMissAnimation(handCard) // shake + dissolve drawFromPool() // still draw replacement // Note: pool is now 1 smaller — fewer cards available to win // 2. Check win/lose conditions after every tap if middleCards.isEmpty: triggerWin() // ALL targets cleared elif isWinnable() == false: triggerLose(reason: "no_matching_cards_left") // unwinnable state // Pool exhausted mid-game drawFromPool(): if pool.isNotEmpty: hand.add(pool.drawTop()) // else: hand shrinks — player finishes with fewer cards
⚠️
Unwinnable state detection is critical. After every tap, check: does the remaining pool + current hand contain at least one card of every colour still in the middle? If not, the game is unwinnable. Trigger a graceful lose state rather than leaving the child stuck — this avoids frustration, which is an EBP concern for this audience.
Win / Lose Conditions
Outcome Trigger Screen
WIN All target cards in middle zone cleared Celebration animation → reward step
LOSE — Time up Timer reaches 0 with targets remaining Gentle "Time's up" → show remaining targets → retry option
LOSE — No matches Pool + hand contains no colour matching any middle card "Oops, no more matching cards" → show what colours were needed → retry
ℹ️
Duplicate colour rule: When the middle zone has two RED cards, tapping a RED hand card clears both simultaneously. This rewards efficient colour recognition. The score multiplier applies — clearing 2 cards with 1 tap scores higher than 2 separate taps would (if that were possible).

Colours are introduced progressively by difficulty. The server specifies which colour set to use in the SDUI config. All colours are bold, high-saturation, and perceptually distinct — chosen for accessibility and therapy efficacy.

This mirrors the DTT errorless-to-errorful progression from the Synergy module spec.

ℹ️
Guaranteed match (Easy): At Easy difficulty, the client must ensure at least one hand card always matches a middle target. This is implemented during pool generation on the server — the pool is seeded so a match always exists. This is the Errorless Learning principle from the EBP spec: early success is essential before introducing failure.
Pool Composition Rules

The server generates the pool and must follow these constraints before sending the SDUI payload:

Rule Easy Medium Hard
Every middle colour must appear ≥ N times in pool 3× 2× 1×
Non-matching distractor colours in pool 0–20% 20–40% Up to 60%
Max duplicate of any single colour in pool 3 5 Unlimited
Pool validated before sending to client ALWAYS — server rejects any unwinnable pool

Scoring & Penalties

// Base score per tap matchScore = basePoints × cardsCleared × speedMultiplier basePoints = 10 // per target card cleared cardsCleared = count of duplicate colours removed // 1 normally, 2-3 if duplicates speedMultiplier = 1.0 (untimed) | 1.0–2.0 (timed, based on time remaining %) // Speed multiplier (timed games only) speedMultiplier = 1.0 + (timeRemaining / totalTime) // e.g. 30s remaining out of 90s → multiplier = 1.33 → score × 1.33 // End-of-round bonus completionBonus = 50 × (1 - (errorCount / totalTaps)) // perfect game = +50 bonus. Every error erodes it. // Final score roundScore = Σ matchScore + completionBonus
Penalty System
Action Consequence Visual Feedback EBP Note
Correct match (single) +10 pts × speed multiplier. Card cleared. Burst animation, card flies up and dissolves Continuous reinforcement
Correct match (×N duplicates) +10 × N pts. All N cards cleared simultaneously. Multi-burst, score pops with ×N indicator Bonus for efficiency
Miss (no middle match) Card permanently lost. Error count +1. No score deducted — loss is the card itself. Card shakes → dissolves with grey tint. Subtle sound. Natural consequence, no punishment sound
3 misses in a row No additional penalty. Parent prompt becomes visible if configured. Parent hint panel slides up Prompt hierarchy trigger
Round complete — no errors Perfect bonus +50 Gold star overlay on celebration Variable reinforcement at mastery
🚫
Never use a negative score. Score never decreases — it only stops growing. Losing a card is sufficient consequence. A decreasing score is psychologically punishing, which violates the EBP constraint. The penalty is opportunity cost (fewer cards to win with), not point deduction.
// 06

Game State Machine

// Complete game state machine — implemented as a Riverpod StateNotifier IDLE onConfigReceived(GameConfig) → LOADING action: validate config, build pool, deal hand, populate middle LOADING onReady() → INTRO action: show card deal animation (cards fly in from top) INTRO onIntroComplete() → PLAYING action: start timer if timed, enable card tap interactions PLAYING onCardTapped(card) [MATCH] → ANIMATING_MATCH onCardTapped(card) [MISS] → ANIMATING_MISS onTimerExpired() → LOSE_TIME onUnwinnable() → LOSE_STUCK ANIMATING_MATCH // ~600ms lock — prevents double-taps onAnimationComplete() [middleEmpty] → WIN [else] → PLAYING action: remove matched targets, draw new hand card, update score ANIMATING_MISS // ~400ms lock onAnimationComplete() [unwinnable] → LOSE_STUCK [else] → PLAYING action: discard hand card, draw replacement, increment errorCount WIN action: stop timer, play celebration animation, compute final score onCelebrationComplete() → RESULTS LOSE_TIME action: freeze board, show "Time's up" overlay onRetryTapped() → LOADING onExitTapped() → IDLE LOSE_STUCK action: show which colours were still needed, gentle message onRetryTapped() → LOADING onExitTapped() → IDLE RESULTS action: POST session result to /sessions, show reward step onResultsAcknowledged() → IDLE
ℹ️
Animation lock states are non-negotiable. ANIMATING_MATCH and ANIMATING_MISS disable all tap interactions. Without this, rapid tapping during an animation causes double-plays, incorrect state, and visual glitches. The lock duration should match the animation duration precisely.

Animation Spec

Duration
Event
Description
0–400ms
Card deal (intro)
Each hand card slides in from bottom with staggered 80ms delay. Each target card drops from top with staggered 60ms delay. Uses flutter_animate's slideY + fadeIn.
0–150ms
Card tap (select)
Hand card scales up to 1.15× on tap-down, back to 1.0 on tap-up. Immediate haptic feedback (HapticFeedback.lightImpact).
0–600ms
Match animation
Hand card launches upward toward matched target card(s) via curved path animation. On arrival: colour burst particle effect radiates outward. Target cards scale up to 1.2× then dissolve with opacity 0. Score popup floats upward (+10, +20 for multi). Uses TickerProvider + CurvedAnimation (easeInOut).
0–400ms
Miss animation
Hand card shakes left-right (3 oscillations, 8dp amplitude). Card then desaturates to greyscale and dissolves downward. NO negative sound — use a neutral muted "whoosh". No red flash or X mark.
0–200ms
Draw new card
New card slides in from right edge of hand zone to fill the gap. Slides in simultaneously with match/miss animation completing — no perceived gap in hand size. Uses SlideTransition from right.
0–100ms
Timer warning
When timer < 10s: timer text pulses red with scale 1.0→1.15 each second. When < 5s: screen edge subtly flashes red on each pulse (not distracting, just peripheral urgency).
0–2000ms
Win celebration
All remaining hand cards fan outward like a deck spread. Lottie confetti/stars animation from reward Lottie URL. Score counter animates from 0 to final score. Celebration respects pref_reduce_animations — falls back to static star display if enabled.

4-Phase Implementation Plan

Phase 1 Core Game Loop — No Animations Days 1–4
  • Implement ColourMatchingNotifier (Riverpod StateNotifier)
    Full state machine with all phases. onCardTapped(), drawFromPool(), checkWinnable() — all pure logic with no UI. Unit test every state transition before writing a single widget.
    LogicState
  • Build ColourCard widget — solid colour rectangle with label
    Stateless widget. Takes a ColourCard and renders: rounded rect, fill colour, white label text (dark for yellow), duplicate badge if N > 1. Respects pref_high_contrast via InheritedWidget.
    UI
  • Build MiddleZone and HandZone layout widgets
    MiddleZone: Wrap widget, cards auto-flow and scale. HandZone: Row with SingleChildScrollView for overflow. Both watch ColourMatchingNotifier and rebuild reactively.
    UIState
  • Build HUD bar — score, pool counter, error count, timer
    Fixed 56px bar at top. Timer only renders when config.timed == true. Pool counter shows X/Total. Error count shown as red X icons (max 3 shown, then a number). All values watch the notifier.
    UI
  • Wire tap handler + instant state update (no animation yet)
    Cards disappear/appear instantly for now. Confirm the full loop works: tap match → middle card removed → hand card removed → new card appears. Tap miss → hand card removed → new card appears → errorCount increments.
    LogicUI
  • Build Win / Lose / Stuck screens (static, no animation)
    Simple overlay widgets for each terminal state. Win: shows score. Lose time: shows "Time's up" + remaining targets. Lose stuck: shows which colours were still needed. All have retry and exit buttons.
    UI
✅
Phase 1 milestone: Fully playable game with no animations. All three difficulty configs work. Winnable state detection prevents getting stuck. State machine tested. Timer counts down correctly on timed mode. The game is functional — just not yet beautiful.
Phase 2 Animations & Haptics Days 5–9
  • Card deal intro animation — staggered slide-in
    Hand cards slide up from bottom with 80ms stagger. Target cards drop from top with 60ms stagger. Uses flutter_animate with slideY + fadeIn. Total intro: ~400ms. Gate tap interactions until intro completes.
    Animation
  • Match animation — card flies to target + burst particle effect
    Use a custom TweenAnimationBuilder with a curved path. Card moves from hand position to target position. On arrival: render a burst of 8–12 small coloured dots that radiate outward using AnimatedWidget. Target cards scale up then fade out. Score popup floats up using OverlayEntry.
    Animation
  • Miss animation — shake + desaturate + dissolve
    Shake: AnimationController driving a Tween<Offset> with a sine curve (3 oscillations). Desaturate: ColorFilter.matrix grayscale transition. Dissolve: opacity 1.0→0.0 with downward translate. Neutral sound only — no negative audio cue.
    AnimationEBP
  • Draw animation — new card slides in from right
    As a hand card is removed, the replacement slides in from the right edge of the hand zone simultaneously. Use AnimatedList with a custom insertion animation. The hand should never visually "jump" — the new card fills the space smoothly.
    Animation
  • Tap scale feedback + haptics on every card tap
    Every card tap triggers: scale 1.0→1.15→1.0 (150ms), HapticFeedback.lightImpact() on tap-down. Match: add HapticFeedback.mediumImpact(). Miss: add HapticFeedback.heavyImpact() (stronger to signal the loss). Respect pref_reduce_animations — disable scale if set.
    AnimationEBP
  • Win celebration — Lottie + score count-up
    Load Lottie URL from on_complete.celebration_config.animation. Score counter animates from 0 to final score over 1.5s using TweenAnimationBuilder. If pref_reduce_animations is true: show static gold star instead of Lottie.
    AnimationEBP
✅
Phase 2 milestone: Game feels polished and satisfying. All animations run without jank. Animation lock prevents double-taps. Match feels rewarding, miss feels neutral-but-clear. Reduce animations setting works throughout. Haptics fire correctly on all interactions.
Phase 3 Difficulty, Timer & Edge Cases Days 10–13
  • Implement countdown timer with visual warning states
    Use a Timer.periodic driven by the state notifier. Below 10s: timer text turns red and pulses. Below 5s: screen edge flashes red. Timer cancels immediately on win. On expiry: transitions to LOSE_TIME state.
    LogicUI
  • Hand scaling for large hand sizes (7–10 cards)
    At 7+ cards, card width shrinks from 64dp to fit within screen width. Use LayoutBuilder to calculate available width, then compute card size: cardWidth = (screenWidth - padding) / handSize. Minimum card width: 44dp. Below 44dp, enable horizontal scroll instead.
    UI
  • Duplicate badge display and multi-clear scoring
    Compute duplicate counts in MiddleZone: group target cards by colour, show ×N badge on each group's representative card. When a matching hand card is tapped, all cards of that colour animate out simultaneously. Score popup shows ×N multiplier.
    LogicUI
  • Pool exhaustion — hand shrinks gracefully
    When the pool is empty and a hand card is played, no replacement is drawn. The hand zone contracts smoothly (remaining cards redistribute with animated spacing). HUD pool counter shows "0" in amber. This is not a lose condition by itself.
    LogicAnimation
✅
Phase 3 milestone: All three difficulty tiers work correctly. Timer runs, warns, and triggers correctly. Large hands scale without overflow. Duplicates clear and score correctly. Pool exhaustion is graceful. All edge cases tested with unit tests.
Phase 4 EBP Integration, SDUI Wiring & Session Reporting Days 14–17
  • Register colour_matching in ComponentRegistry
    Add 'colour_matching': (config, session) => ColourMatchingGame(config: config, session: session) to the Flutter SDUI component registry. The game now receives its full config from the server without any hardcoded values in the widget.
    SDUI
  • Prompt hierarchy — parent hint panel at 3 consecutive misses
    When consecutiveMisses >= 3 AND config.promptLevel >= 3: a collapsible parent hint panel slides up from below the hand zone showing the parent_instruction text from the SDUI config. Parent can dismiss it. This is the in-game embodiment of the prompt hierarchy.
    EBPUI
  • Session result POST on game complete
    On WIN or LOSE: build the session result object (correct_matches, misses, cards_cleared, pool_remaining, time_used_seconds, outcome) and POST to /sessions via the SessionController. Handle network errors gracefully — queue locally and retry if offline.
    LogicState
  • Audio integration — match sound, miss sound, timer warning, celebration
    Integrate with settingsProvider: check sfxEnabled and volume before playing any sound. Match: bright chime. Miss: neutral whoosh (no negative sound). Timer warning: soft ticking below 10s. Celebration: fanfare. All sounds muted if sfxEnabled == false.
    EBPLogic
